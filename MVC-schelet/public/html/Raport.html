Autori
Birleanu Rebeca , Olteanu Dragos si Lupancu Camelia Viorica

Introducere

Aplicația web CompIT ofera informații cu privire la câteva categorii de produse ,preluate din surse multiple, din care utilizatorul logat sau nu poate sa aleaga cu scopul de a obține studii comparative pentru acele produse pe baza anumitor criterii universale precum pret sau rating, dar și specifice fiecarei categorii.

Motivație

Aceata aplicație web reprezintă proiectul pe care noi am ales sa il abordam in cadrul cursul Tehnologii Web al Facultății de Informatică al Universității Alexandru Ioan Cuza.

Cerința proiectului

Pe baza unui API REST ori GraphQL propriu, realizati o aplicatie Web – disponibila, de asemenea, ca extensie de navigator Web– care furnizeaza utilizatorilor autentificati sau nu studii comparative privitoare la (fluctuatia de) preturi ori alte caracteristici – e.g., model mai recent, varianta similara etc. – asociate unor articole/servicii apartinand unei/unor categorii de interes (electrocasnice, incaltaminte sport, API-uri de recomandare etc.). Datele analizate vor fi preluate din surse multiple (fluxuri de stiri, API-uri disponibile, via scraping - minim 3) oferite de situri de profil ce vor putea fi precizate de utilizator. Recomandarile generate vor fi disponibile si sub forma de fluxuri de stiri RSS.

Procesul de dezvoltare

Dezvoltare pe partea de client

Prima etapa a fost cea a dezvoltarii parții de Front-End . Am inceput prin a gandi cum va arata aplicația, care vor fi paginilie, care vor fi culorile principale etc. Am trecut la implementarea continutul static de HTML și CSS de baza, mai apoi pe parcursul dezvoltarii aplicației adaugând și alte elemente in funcție necesitați.

Paginile principale pe care le-am construit de la început au fost: home, login/register, pagnia pentru afisarea produselor, o pagina pentru contactarea administrației si o pagina de prezentare a misiunii aplicației.

In dezvolatarea acestora ma cautat sa abordam designul Web responsiv prin diferite metode prezentate la laborator (media queries, folosirea procentelor in definirea de proprietați ca width ). Paginile au fost analizate să fie valide în conformitate cu standardele. Au fost folosite tool-uri pentru validare de html și de css.

Implementarea aplicației pe partea de server

Structura de MVC

Dupa ce am avut pagnile de baza gata primul lucru pe care l-am facut a fost sa contruim serverul dupa modeul MVC care sa servearca pagnilie la cerere. Pentru structura de MVC am realizat trei clase principale Controller, View si Model pentru generalizare care urmau sa fie extinse de controllere-le si modelele concrete. Pentru rezolvarea cererilor am realizat clasa Application care functioneaza ca un router fiind responsabila cu apelarea acțiunii potrivite din controller-ul potrivit.

A urmat impementarilor serviciilor de creare de cont, de logare in cont si de contact la administrației. Pentru acestea am folosit o baza de date locala oferita de XAMPP (myPHPAdmin). Aceasta foloseste MySQL, foarte usor de folosit cu PHP. Functiile care apeleaza baza de date sunt puse in clase care extind clasa Model și sunt apelate din clase care extind calsa Controller.

Conectarea cu Google

Un alt serviciu pe care l-am dezvoltat este cel de conectarea cu contul de Google. Pentru aceasta, cu ajutorul managerului de dependețe pentru PHP Composer am instalat biblioteca google/apiclient. De asemeneam am creat un cont pentru plicația noastra pe Google Developers pentru a obține token-ul pentru clasa Google_Client.

eBay

Am început sa ne documentam cu privire la sursele de unde aveam sa preluăm produsele. Prima sursa abordata a fost API-ul oferit de eBay pentru a prelua produse din diferite categorii. Cu ajutorul acestui API am implementat și serviciul de cautare a unui produs dupa cuvinte cheie oferit de pagina principala.

In cererea trimisa prin metoda GET de fiecre data cand se cauta un produs sunt specificați in query string cativa parametrii precum appid (id-ul din contul de eBay developers ), QueryKeywords (cuvintele cheiedupa care se face cautatea), AvailableItemsOnly ( boolean ), MaxEntries ( numarul de produse returnate). Raspunsul este in format XML cu taguri specifice. Raspunsul l-am parsat transformadu-l intro-o lista de obiecte de tipul Product și le-am afisat în client. Aceste date nu sunt memorate in baza de date. Daca utilizatorul alege un produs pe partea de client sunt prelurat in javaScript informațiile afișate în pagina și trimis la server printr-o cerere facuta cu ajutorului thehnologiei AJAX .

Preluarea informațiilor prin scarping

Tot pe partea de preluare de informații privitoare la categoriile alese de noi, am ales sa facem scraping de pe situri de specialitate (Emag, Altex, CEL). Am stocat inforații genrice în baza de date locala deoarece sa am realizat ca sa facem scraping dinamic la fiacre incarcare de pagina dureaza destul de mult. Pentru consistența am decis ca baza de date sa fie updata constant. Din cauza ca scraping-ul conuma timp am decis să nu updatam toate produsele din baza de date data, ci atunci cand un utilizator alege un produs se updateaza doar produsul respectiv.

Pentru scraping am utilizat biblioteca cu licenta deschisa simple_html_dom care permite manipularea sursei HTML mai ușor pentru gasirea etichetelor si extragerea conținutului acestora.

Al doilea server

Pentru a nu incarca foarte mult baza de date si pentru a distribui munca, am constuit un la doilea server care sa se ocupe excusiv cu managerierea produselor alese de utilizatori pentru comparare. Sub forma uni API REST, am expus posibilitatea ca utilizatorii cu cont sa trimita direct poroduse pentru a le compara. Ei sunt recunoscuti pe baza unei chei oferite in cont. Acest server se ocupa si cu scraping-ul pentru detaliile produselor. Pentru stocarea datelor se foloseste o baze de date aliniate paradigmei NoSQL in cloud, MongoDb.

Extensie de navigator Web

Cu ajutorul fisierului manijest.json am facut aplicatia disponibila ca extensie de browser.

Modulul de administrare

Am inceput prin a face o interfata pe partea de client pentru administratorul aplicatiei. Acesta se conecteaza cu o parola specifica . I se ofera accesul la baza de date a aplicației, poaye vizualiza mesajele lasate de utilizatori prin formularul de contact (poate alege sa le vizualizeze pe toate sau doar pe cele noi). De asemnenea, poate schimba parola contului de administrator.

Am realizat și un fisier de configurare app.ini in care sunt pastrate informații de acces la baza de date, portul la care ruleaza cele doua servere, sursele din care provin produsele, care surse sunt active și ale infomrații necesare aplicației care pot fi configurate.

Unit Testing

Pentru partea de tastare am apelat la PHPUnit care este un cadru de testare pentru limbajul de programare PHP. Am scris teste pentru metodele din cadrul claselor Model, testând in felul acesta atat partea de citrie și scriere a datelor din baza de date, cat și logica acestora. Deoarece funcțiile din modele fac trimtit cereri la al doiela server am testat si funcționalitatea API-ului acestuia. Se verifica atat tipul intors, cat si valoarea returnata prin funcțiile de assert.

End-Ponit: /AddProduct

Verbul HTTP folosit este POST. Ruta aceasta este utilizata pentru adaugarea unui produs nou la lista de comparare a utilizatorului care face cererea. Cerea trebuie sa contina un body de tip json care sa aiba urmatoarele campuri : "category" ,"title", "link","img_link", "source", "details", "price"(va fi meționat in lei ).

End-Ponit: /UpdatePrice

Verbul HTTP folosit este PUT. Ruta aceasta este utilizata pentru update la pretului unui produs din lista de comparare a utilizatorului. Cererea trebuie sa fie insotita de un body de tip json care sa conțina campurile "id" (id-ul produsului primit cand produsul este adugat) si "new_price".

End-Ponit: /DeleteProduct?id=id-ul_produsului

Verbul HTTP folosit este DELETE. Ruta aceasta este utilizata pentru stergerea unui anumit produs din lista de comparare a utilizatorului. În query sting se competa cu parametru id care este id-ul unic al produsului care va fi sters.

End-Ponit: /GetMyProducts

Verbul HTTP folosit este GET. Ruta aceasta este utilizata pentru returnarea tuturor produselor din lista utilizator. Utilizatorul va fi recunoscut pe v=baza campului "Session" din Heardes.

Arhitectura aplicatiei

Tehnologii și limbaje

Limbajul de programare pentru server

Am decis sa folosim PHP pentru implementarea serverului deoarece este un limbaj optimp pentru devoltarea aplicațiilor web fara folosirea frameworkurilor.

Pentru instalarea de biblioteci externe am apelat la Composer. Cu care am integrat bibliotecile de care am avut nevoie.

Limbajul pe partea de client

Pe langa HMTL și CSS, pe partea de client am utilizat si JavaScript pentru anumite elemente mai dinamice din pagina, dar si pentru a trimtie informații la server și a încarca informații de pe server fara a reîncarca sau schimba pagnia.

Github

Pentru dezvoltarea proiectului in echpa am utilizat Github care reprezinta o varianta pentru lucrul in echipa la distanța.

Baza de date

Am folosit baza de date oferita de XAMPP (phpMyAdmin) pentru serverul principal deoarece e foarte usor de utilizat in combinație cu limbajul PHP . Pentru memorarea produselor, preluate din surse diverse, pe serverul la doilea am optat pentru o varinata de baze de date NoSQL, MongoDb, deoarece ne oferea mai multa flexibilitatea in memorarea documenteleor.
